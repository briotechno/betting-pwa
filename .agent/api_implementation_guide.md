# 📄 Betting PWA API Implementation Guide

This guide details the implementation of the Betting Platform API in this project. It is designed to be a blueprint for migrating the API layer to other projects.

---

## 🏗️ 1. Architecture Overview

The API layer is centralized for security, maintainability, and consistent error handling.

### 📁 Directory Structure
```text
src/
├── controllers/          # Business logic & individual module endpoints
│   ├── auth/             # Login, Signup, OTP
│   ├── user/             # Profile, Balance, Settings
│   ├── wallet/           # Deposits, Withdrawals, Statements
│   ├── market/           # Sports, Games, Competition Data
│   ├── betting/          # Placing bets (Odds, Fancy, Bookmaker)
│   ├── casino/           # Casino & Sportsbook launchers
│   └── index.ts          # Central export hub
├── utils/
│   └── api.ts            # Core logic (Hashing, Base URL, Fetch wrapper)
└── store/
    └── authStore.ts      # Session management (Zustand)
```

## 🔐 2. Security & Protocol

### 🛂 Authentication (HMAC SHA-256)
Every request to the API requires a `Hash` header. This hash is generated by signing the JSON stringified request body with a shared `API_SECRET`.

**Implementation details (`src/utils/api.ts`):**
- **Algorithm**: HMAC-SHA256
- **Encoding**: Base64
- **Secret Provider**: `process.env.NEXT_PUBLIC_API_SECRET`

```typescript
// Example Hashing Logic (using crypto-js)
const hash = CryptoJS.HmacSHA256(JSON.stringify(body), API_SECRET).toString(CryptoJS.enc.Base64);
```

### 🛰️ Base Infrastructure
- **Base URL:** `https://ambikaexch.in/extsys`
- **Protocol:** Always `POST` for both fetching and mutations.
- **Headers:**
  - `Content-Type: application/json`
  - `Hash: <computed_signature>`
- **Response Format:**
  - `error: "0"` -> Success
  - `error: "1"` -> Failure (message in `msg` field)

## 🛠️ 4. Handling Implementation "Quirks"

During implementation, we identified several protocol nuances that must be followed in another project:

1.  **Reused Endpoints**:
    - `/dealextra` is used for **both** Extra markets and Goal markets.
    - `/offersdetail` is used for **both** viewing details and claiming the offer.
2.  **Key Naming Anomalies**:
    - `/statementbet` requires a leading space in the key: `{" Eid ": "ID"}`.
    - `/createuser` might require a leading space in some environments: `{" mobile": "..."}`.
3.  **Data Formats**:
    - **Odds**: Usually provided in decimals (e.g., 1.50).
    - **Fancy**: Market odds for "Yes" and "No" are separate and require `No` and `Yes` parameters even when placing a single side.
    - **Base64**: Most assets (Screenshots, Popups, Banners) are returned as raw Base64 strings or HTML strings containing Base64 tags (e.g., `<img src="...">`).

## 🚀 5. Migration Guide (Checklist)

To use these APIs in a new project:

1.  **Dependency**: Install `crypto-js` for hashing.
2.  **Environment Variables**:
    ```env
    NEXT_PUBLIC_API_URL=https://ambikaexch.in/extsys
    NEXT_PUBLIC_API_SECRET=F41985E0-D500-4E68-AF71-3701EFC9637A
    ```
3.  **Core Utility**: Copy `src/utils/api.ts` to your new project. This handles the hashing and JSON transformations.
4.  **Controllers**: Copy the `src/controllers` directory. These are purely functional and rely on `fetchAPI`.
5.  **Headers**: Ensure `Hash` is always capitalized exactly as "Hash" in headers.
6.  **Session Hook**: Replicate `useAuthStore` or a similar store to manage `LoginToken`. Pass this token to almost every authenticated endpoint as `LoginToken`.

---

> [!IMPORTANT]
> **Normalization Rule**: All responses should be checked for `error === "0"` before proceeding. If `error === "1"`, user feedback should be triggered using the `msg` field.
